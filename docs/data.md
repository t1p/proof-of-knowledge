# Сущности и транзакции системы Protocol Attestation

> **Note**: Техническая реализация в смарт-контрактах сохраняет legacy naming (`ValidatorContract`, `Validation`). Данный документ описывает сущности в терминологии Protocol Attestation.

> **Legacy naming mapping**: 
> - `validation` → `attestation` (system output)
> - `verification` → `verdict` (community-issued conclusion)
> - `validator` → `attestor` (person issuing attestation)
> - `validation result` → `verdict` (final community-interpretable outcome)

> **Разграничение терминологии**:
> - **Attestation** (аттестация) — технический вывод системы: криптографически подписанная запись о соответствии объекта правилам протокола.
> - **Verdict** (вердикт) — смысловая интерпретация аттестации сообществом: «сообщество проверило, что объект соответствует протоколу X».
> - **Attestor** (аттестор) — участник, выполняющий аттестацию согласно протоколу сообщества.
> - **Applicant** (заявитель) — участник, подающий объект на аттестацию.

## Attestation Contracts (сущности системы)

### Создание сообщества (Community)

1. Инициатор сообщества — публичный ключ
2. Название сообщества — поле text 256?
3. Область аттестации — имена из классификатора OpenData
4. Ссылка (+хеш) на описание и вступление в сообщества в IPFS
5. Залог за присоединение к сообществу — сумма 5 у.е.
6. Залог за создание сообщества — сумма 25 у.е.
7. Токен — Название / тикет

Дополнительно Инициатор в момент создания (может быть изменено голосованием в будущем) принимает два обязательных решения (отдельными транзакциями):

#### Параметры голосования:

1. Длительность голосования — часы дни и тд.
2. Требуемый Кворум (от 100 до 0%, т.е. даже без голосования)

#### Критерии истиности / правила аттестации / методика

1. Ссылка на методику — URL IPFS
2. Стоимость аттестации (минимальной) — N токенов сети
3. Минимальное кол-во участников аттестации (def 3)
4. Время реакции на предложение (def 24 h)

**Результат** — сообщество (DAO) создано. Задеплоен контракт сообщества. В нем видно кто создал и его свойства. В базу сообществ отправлен УИД и область.

### Вступление в сообщество (AttestorJoining)

1. УИД сообщества — ИД смарт контракта сообщества
2. Кандидат — публичный ключ
3. Залог — N токенов сети
4. Пользовательские данные (пример: имя кандидата, ссылка на профиль соц сетей, документ, etc)

**Результат** — Кандидат стал участником сообщества (attestor). В блокчейне есть запись о новом участнике.

### Предложение голосования участником (AttestorProposal)

Как для изменения обязательных решений, так и для добавления новых.

1. Участник — публичный ключ
2. Ссылка на описание предложения — URL IPFS
3. Ссылка (опция) на предыдущее голосование — УИД голосования. Если выбрана, то указываются действие (изменить, удалить) и новые атрибуты предлагаемого решения

**Результат** — Предложение в блокчейне со статусом "на голосовании". Участники могут голосовать. Если набран кворум, голосование останавливается до окончания срока. Статус "принято". Если кворум набран не был, то статус не "принято". Если принятое решение затрагивает принятое ранее решение, то прошлая версия — статус "не действует", новая — "действует" с измененными атрибутами.

## Applicant Contracts (взаимодействие с пользователем)

### Предложение на аттестацию (AttestationRequest)

1. Заявитель (Applicant) — публичный ключ
2. Ссылка с данными на аттестацию — URL IPFS
3. Предлагаемая цена аттестации (она же залог за аттестацию) — N токенов сети
4. Параметры аттестации (срок, кол-во аттесторов, etc)

**Результат** — участники получают оповещение о заявке на аттестацию (в метамаске или др.).

### Согласие аттестировать Предложение (AttestationBid)

1. Участник — публичный ключ
2. Залог за аттестацию — N токенов сети
3. Условия приняты — факт транзакции

**Результат** — При достижении минимального кол-ва участников, участники могут отправлять в блокчейн свои решения по аттестации.

### Результат аттестации предложения (Verdict)

1. Участник (Attestor) — публичный ключ
2. Решение — валид, не валид, не применимо (скрыто от паблика)
3. Ссылка на протокол аттестации (опция) — URL IPFS

**Результат** — При получении результатов от всех аттесторов, сеть сравнивает их результат. Если 2/3 аттесторов дали одинаковое решение, сеть генерирует по 3 х N токенов аттесторам, и 1,5 х N заявителю. Также заносит +1 в поле репутации аттесторов и +1 заявителю успешные аттестации. Также присваивается "+1 валид" информации по данному адресу и она кешируется в IPFS нодах сети. Если 2/3 аттесторов дали различные решения, то залоги не возвращаются. В неуспешные аттестации аттесторам и заявителю -1.

## Терминологическое разграничение: Documentation vs. Implementation

| Понятие в документации (Protocol Attestation) | Существующее имя в контрактах (Legacy) |
|-----------------------------------------------|--------------------------------------|
| Attestation | Validation |
| Verdict | ValidationResult |
| Attestor | Validator |
| Applicant | User |
| Community | DAO / ValidatorCommunity |
| AttestationRequest | ValidationRequest |
| AttestationBid | ValidatorBid |
